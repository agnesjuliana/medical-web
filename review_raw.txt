Great work implementing the Onboarding Results screen! The UI matches the specifications well and the components are nicely structured. However, I found a few critical logic bugs regarding unit conversions and some dead code that needs to be cleaned up.

### 1. Bug: Imperial Unit Parsing Failure in TDEE & Goal Calculation
**File**: `modules/modul_8/app/src/components/page/OnboardingPage.tsx`

**Issue**: In `computeTDEE()`, `computeGoalDate()`, `MotivationContent`, and inside the `OnboardingPage` render block, `parseInt(form.weight)` and `parseInt(form.height)` are used. If the user toggles the unit to Imperial, `form.weight` will be a string like `"154 lbs"` and `form.height` will be `"5'7""`. Calling `parseInt("5'7\"")` evaluates to `5` (centimeters!), and `parseInt("154 lbs")` evaluates to `154` (kilograms!). This leads to wildly incorrect BMR calculations and calorie targets.

**Fix**: Create helper functions to safely parse these values into metric equivalents before computation:

```typescript
// Add these helpers
function parseHeightCm(heightStr: string): number {
  if (!heightStr) return 170;
  if (heightStr.includes("'")) {
    const [feet, inches] = heightStr.split("'").map(s => parseInt(s) || 0);
    return Math.round((feet * 30.48) + (inches * 2.54));
  }
  return parseInt(heightStr) || 170;
}

function parseWeightKg(weightStr: string): number {
  if (!weightStr) return 70;
  if (weightStr.includes("lbs")) {
    return Math.round((parseInt(weightStr) || 154) / 2.20462);
  }
  return parseInt(weightStr) || 70;
}
```
Then replace `parseInt(form.weight)` with `parseWeightKg(form.weight)` and `parseInt(form.height)` with `parseHeightCm(form.height)` everywhere these values are used.

### 2. UX/Logic: Mixed Units Display on Results Page
**File**: `modules/modul_8/app/src/components/page/OnboardingPage.tsx`

**Issue**: The `plan` object sets `goalLbs` using a hardcoded pounds conversion (`parseFloat((diff * 2.20462).toFixed(1))`), which `OnboardingResults` always displays as `lbs`. However, the goal item list below it hardcodes `kg` (`title: \`${action} ${diff} kg\``). This creates a confusing UX where the user sees both pounds and kilograms on the same screen, regardless of their selected preference.

**Fix**: Pass a dynamic `targetWeight` and `targetUnit` to the `plan` object based on whether the user selected Metric or Imperial, and update `OnboardingResults.tsx` to use those props instead of hardcoding `lbs` and `kg`.

### 4. Potential Bug: URL-encoded Passwords in Connection String
**File**: `config/database.php`

**Issue**: `parse_url()` does not automatically URL-decode the `user` and `pass` components. If the `DATABASE_URL` contains special characters in the password (e.g., `postgres://user:p%40ssword@host...`), the connection will fail because `p%40ssword` will be passed to PDO instead of the actual password `p@ssword`.

**Fix**: Wrap the parsed credentials in `urldecode()`.

```php
-    define('DB_USER', $parsed['user'] ?? 'postgres');
-    define('DB_PASS', $parsed['pass'] ?? '');
+    define('DB_USER', urldecode($parsed['user'] ?? 'postgres'));
+    define('DB_PASS', urldecode($parsed['pass'] ?? ''));
```

Please address these points. Let me know if you have any questions!
