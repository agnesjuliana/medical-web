Reviewed commit `90d4162999ba5afb93c319a31586ec5886dcd4d6` only. I found a few issues that should be fixed before merge.

### 1) 🔴 Blocking: AI quota check is no longer atomic (race condition)
**Location:** `modules/modul_8/Backend/Controllers/AiController.php:48-51,81-82` and `modules/modul_8/Backend/Repositories/AiScanRepository.php:32-58`

Current flow does `canScanToday()` then later `incrementScanCount()`. Under concurrent requests, multiple scans can pass the check and all increment, allowing quota to exceed the daily limit.

**Suggested fix:** keep increment-after-success behavior, but make the increment itself conditional+atomic.

```php
// Repository
public function incrementIfUnderLimit(int $userId, int $limit): bool
{
    $stmt = $this->pdo->prepare('
        INSERT INTO m8_ai_scan_quota (user_id, log_date, scan_count, created_at, updated_at)
        VALUES (?, CURRENT_DATE, 1, NOW(), NOW())
        ON CONFLICT (user_id, log_date) DO UPDATE
            SET scan_count = m8_ai_scan_quota.scan_count + 1,
                updated_at = NOW()
        WHERE m8_ai_scan_quota.scan_count < ?
        RETURNING scan_count
    ');
    $stmt->execute([$userId, $limit]);
    return (bool) $stmt->fetchColumn();
}

// Controller (after successful parse/sanitize)
if ('18,80p'this->repo->incrementIfUnderLimit($userId, $limit)) {
    $this->jsonError("Daily AI scan limit reached ({$limit}/day)", 429);
}
```

### 2) 🔴 Blocking: `log_date` validation can produce 500 and accepts impossible dates
**Location:** `modules/modul_8/Backend/Controllers/MealController.php:57-63`

`preg_match` only checks shape, then `new DateTimeImmutable($log_date)` is used directly.
- Some invalid inputs throw exception (`2026-99-99`) and bubble as 500.
- Some impossible dates normalize silently (`2026-02-31` becomes `2026-03-03`).

**Suggested fix:** strict date parse with format round-trip check.

```php
$logDateObj = \DateTimeImmutable::createFromFormat('Y-m-d', $log_date);
if ('18,80p'logDateObj || $logDateObj->format('Y-m-d') !== $log_date) {
    $this->jsonError('Invalid log_date', 422);
}
if ($logDateObj > new \DateTimeImmutable('today')) {
    $this->jsonError('log_date cannot be in the future', 422);
}
```

### 3) 🟡 Important: goal progress logic fails for weight-gain goals
**Location:** `modules/modul_8/Backend/Controllers/ProgressController.php:75-79`

Current formula sets progress to `0.0` whenever `goalWeight > startWeight` (gain scenario), so users with gain goals never see progress.

**Suggested fix:** compute direction-aware progress for both lose and gain goals.

```php
$goalProgress = 0.0;
if ($goalWeight !== null && $startWeight !== $goalWeight) {
    $direction = $goalWeight > $startWeight ? 1 : -1;
    $numerator = ($currentWeight - $startWeight) * $direction;
    $denominator = abs($goalWeight - $startWeight);
    $goalProgress = max(0, min(100, round(($numerator / $denominator) * 100, 1)));
}
```

### 4) 🟡 Important: JPEG enforcement and `media_type` can become inconsistent
**Location:** `modules/modul_8/Backend/Services/AiScanService.php:37-45`

The code validates JPEG magic bytes, but still forwards `media_type` from user input. A payload like `data:image/png;base64,<jpeg-bytes>` passes signature check yet sends `image/png` upstream, which can cause avoidable API failures.

**Suggested fix:** enforce JPEG MIME explicitly when returning parsed image metadata.

```php
if (!preg_match('/^data:image\/(jpeg|jpg)(?:;[^,]+)*;base64,/', $imageB64)) {
    throw new \InvalidArgumentException('Only JPEG images are allowed');
}
$mediaType = 'image/jpeg';
```
