## Objective
Reduce Home screen latency by optimizing the `get_dashboard` backend hot path. The current implementation performs multiple database reads plus a write on every dashboard load. This task should consolidate the data access pattern so the endpoint can serve the full dashboard payload with fewer database round trips and without write-on-read behavior.

## Target Files to Modify/Create
- `modules/modul_8/Backend/Controllers/DashboardController.php`
- `modules/modul_8/Backend/Repositories/MealRepository.php`
- `modules/modul_8/Backend/Repositories/WaterRepository.php`
- `modules/modul_8/Backend/Repositories/HealthScoreRepository.php`
- `modules/modul_8/tests/Integration/DashboardControllerTest.php`
- `modules/modul_8/schema.sql` if a new supporting index is required

## Step-by-Step Implementation (Low-Level)
1. **Inspect current dashboard flow**: Open `modules/modul_8/Backend/Controllers/DashboardController.php` and identify every repository call inside `getDashboard()`. Confirm that the controller currently calls the profile repository, meal aggregate query, water aggregate query, recent meals query, and health-score upsert.
2. **Add one repository method for dashboard data**: In `modules/modul_8/Backend/Repositories/MealRepository.php`, create a new method named `getDashboardSummaryByDate(int $userId, string $date): array`. This method must return one array containing:
   - `meal_totals` with calories, protein, carbs, fats, fiber, sugar, sodium
   - `recent_meals` limited to the latest 5 rows for that `user_id` and `log_date`
   - `water_ml` for the same `user_id` and `log_date`
   Use PostgreSQL SQL that groups or aggregates the meal rows once and fetches recent meals in the same method. If one SQL statement becomes too hard to maintain, keep it to at most 2 SQL statements inside this method, not 3 separate controller-level calls.
3. **Align query with sort pattern**: If the new summary query still depends on `ORDER BY created_at DESC LIMIT 5`, add a composite index in `modules/modul_8/schema.sql` for `m8_meals (user_id, log_date, created_at DESC)`. Keep the existing index unless it is clearly redundant and safe to replace.
4. **Stop writing health scores during dashboard reads**: In `modules/modul_8/Backend/Controllers/DashboardController.php`, remove the call that upserts into `m8_daily_health_scores` during `getDashboard()`. Keep in-memory health score computation so the API response still includes `health_score`, but do not write to the database from this read endpoint.
5. **Refactor controller to use the new summary method**: In `getDashboard()`, keep profile loading as-is for now, but replace the separate calls to `getAggregatesByDate()`, `getTotalByDate()`, and `getRecentByDate()` with the new summary method. Map its return value into the existing response keys: `targets`, `consumed`, `remaining`, `recent_meals`, and `health_score`.
6. **Preserve response shape**: Do not rename API response keys that the frontend already expects. The output JSON must still contain `date`, `targets`, `consumed`, `remaining`, `recent_meals`, and `health_score`.
7. **Remove dead repository dependency if unused**: After refactoring, check whether `WaterRepository` is still needed by `DashboardController`. If it is no longer used, remove the constructor dependency and update the controller creation in `modules/modul_8/Backend/index.php` so the dashboard controller is not passed an unused repository.
8. **Update tests**: Open `modules/modul_8/tests/Integration/DashboardControllerTest.php` and update or add tests so they verify:
   - the dashboard still returns the same response structure,
   - health score is returned in the response,
   - the dashboard path no longer depends on a health-score upsert side effect to pass,
   - the recent meals list still contains the newest entries first.
9. **Run targeted verification**: Run the Module 8 dashboard integration tests only. If there is a project-standard command for this test file, use it. If not, run the smallest available test command that exercises `DashboardControllerTest.php`.

## Acceptance Criteria
- [ ] `GET /modules/modul_8/Backend/index.php?action=get_dashboard` still returns the same response shape used by the frontend.
- [ ] `get_dashboard` no longer performs a database write to `m8_daily_health_scores` during a normal read request.
- [ ] Dashboard data fetching uses at most one profile query plus one summary repository call for meals/water/recent-meals data.
- [ ] Recent meals are still ordered by newest `created_at` first and limited to 5 rows.
- [ ] Integration tests for dashboard behavior pass after the refactor.
